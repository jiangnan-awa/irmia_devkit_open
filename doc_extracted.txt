装工具箱类项目里，codegraph 的完成度属于头部水平。以下逐项说明出彩之处与现存瑕疵。2.1  优点自研语义索引引擎自己实现了 Python AST + SQLite FTS5 + BFS 调用链的完整管线，不是简单地 grep 封装。三阶段引用解析（Phase 1 短名唯一匹配 → Phase 2 跨文件 import 拼接 → Phase 3 self.method 类内解析）覆盖了绝大多数 Python 常见写法。结构化查询响应设计合理所有 code_* 工具的失败返回都包含 proposal / options / next_call 字段，不是甩裸错误文本，而是给 LLM 一个明确的(下一步该怎么做)。BFS 未找到完整路径时回退到 break_at + break_reason，引导 LLM 用 rg_search 补充确认。增量索引设计正确基于 mtime 的增量索引逻辑正确覆盖了"文件未改则跳过、文件已改则先删后插"的场景（全量 FTS5 重建保证一致性）。主体逻辑无误，缺陷见 §3.3。tree-sitter 多语言可选扩展7 种语言的 grammar 均为可选依赖，零依赖时自动降级到纯 Python 模式，code_status 工具如实反馈哪些 grammar 缺失并给出 pip install 提示。测试覆盖关键路径29 个 codegraph 专项测试用例覆盖了 FTS5 查询、BFS 路径、引用解析、pack/diff_impact/status 等核心分支，且用真实文件系统而非过度 mock。2.2  瑕疵概览（评测前 / 评测后已更正）正式评测发现 2 个 P0 Bug（脚本实测复现）、1 个附带缺陷，以及 1 处措辞不精确（见 §5 自我复核）。详见第 3 节。3  已确认 tools/_registry.py 中，_code_index(project_dir, ...) 和 _code_explore(query, project_dir=".") 均接收并使用调用方传入的 project_dir 来定位 .codegraph/codegraph.db；但 _code_pack、_code_diff_impact、_code_status 这三个 wrapper 函数均硬编码 Path(".").resolve()，即 AstrBot 主进程的当前工作目录，与 code_index 时指定的项目目录完全独立。这三个工具的 Tool 类 parameters schema 里也均无 project_dir 字段，LLM 无法传入正确路径。实测复现以下脚本模拟"在 /projects/myapp 建好索引，但进程 cwd 在 /bot_root"的场景（AstrBot 真实运行时的标准情形）：code_index(project_dir='/projects/myapp')     # 索引建在 /projects/myapp/.codegraph/   → 索引完成：1 文件, 2 符号, 1 边code_explore('foo', project_dir='/projects/myapp')  # found: True# 进程 cwd 切至 /bot_rootcode_status()     # → files_indexed: 0, symbols_total: 0  ← 错误code_pack('foo')  # → {'ok': False, 'error': "符号 'foo' 未找到"}  ← 错误更隐蔽的是：code_status / code_pack 在数据库不存在时会在 cwd 下新建空库（因 _ensure_db 含 CREATE TABLE + os.makedirs），导致"无索引"与"索引为空"两种状态无法区分，LLM 收到 files_indexed: 0 时完全没有线索说明真正原因是路径错误。修复建议短期：给 code_pack / code_diff_impact / code_status 补上 project_dir 参数（默认 "."），与 code_explore 对齐，description 里注明"须与 code_index 时传入的目录一致"。长期（更稳健）：在 index() 写入时把绝对根路径存入 meta 表（meta['root'] = str(root)），打开 DB 后先校验调用方传入路径与 meta['root'] 是否一致；不一致时返回带明确 proposal 的结构化错误，而不是静默查空库。附加：在打开 DB 前先 Path(db_path).exists() 判断，区分"数据库不存在"与"数据库为空"两种状态。3.2  BUG P0-2：源码双重截断 / source_truncated 静默误报问题描述_extract_python 在写入 DB 时把每个函数的 source 截为 [:500] 字符（FunctionDef 第 768 行、AsyncFunctionDef 第 792 行）。读取时 _smart_truncate 再对已截断的残片按 30 行做 head+tail 处理——但此时它看到的已是残缺数据，若残片不超过 30 行则返回 source_truncated: False、total_lines 为残片行数，形成静默误报。code_explore 路径（经由 _row_to_dict）额外存在一个问题：_smart_truncate 返回的 source_truncated 和 total_lines 两个字段根本未被 _row_to_dict 写入返回 dict，LLM 拿到的结果里这两个字段完全缺失。实测复现（42 行函数）项目预期实际原始函数行数42 行存入 DB 的 source 行数42 行（完整）存入 DB 的 source 字符数≤ 2,610 字符（完整）实际存入 DB—code_pack 返回 source_truncatedTrueFalse ✗code_pack 返回 total_lines429 ✗code_explore 返回 source_truncated 字段存在缺失 ✗此问题直接削弱 code_pack"精准上下文打包"的核心价值：LLM 收到 source_truncated: False 会认为源码完整，基于残缺代码生成的 multi_edit.old 字符串可能与文件实际内容不匹配，导致 patch 失败。修复建议根本修复：将 _extract_python 里的 [:500] 截断上限大幅提高至 6,000–8,000 字符（覆盖绝大多数函数完整体），把真正的"按行 head+tail 摘要"完全交给读取时的 _smart_truncate 执行。_row_to_dict 补全字段：把 _smart_truncate 返回的 source_truncated 和 total_lines 透传到 _row_to_dict 的结果 dict 中。截断时附带导航提示：source_truncated: True 时附加 hint 字段，告知 LLM 用 file_read 读取 file 的 line 至 line+total_lines 行获取完整源码；或给 code_pack 增加 full_source: true 选项，对 target 本身不截断。3.3  缺陷 P0-3：增量索引不清理已删除文件的残留符号问题描述增量索引（incremental=True）只遍历磁盘上当前存在的文件，对已删除文件：mtimes 字典会在下次遍历时自然不包含它（因为遍历从磁盘文件集合出发），但 symbols 和 edges 表里属于该文件的行会永久残留。实测复现删除 b.py → 运行 incremental 索引 → code_explore('func_in_b') 仍然返回 found: True，且 file 字段指向一个不存在的路径。mtimes 里 b.py 已清除，但 DB 里的符号未清除，两者不同步。修复建议在文件遍历开始前取 indexed_files = set(mtimes.keys())，遍历结束后计算 deleted = indexed_files - current_files，对差集每个路径执行 DELETE FROM symbols/edges WHERE file=? 并从 mtimes 里移除。4  迭代改进计划（P1–P4）4.1  P1  引用解析增强4.1.1  继承链解析缺失_resolve_references Phase 3 只做同类方法解析（cls_prefix.method），不沿继承链向上追踪。若 class B(A) 且 B 未重写 foo，则 b.foo() 永远解析不到 A.foo；extends 边的 to_sym（如 bar.BaseClass）也不经过任何解析阶段，无法连接到真正的符号定义。改进方向：利用已有 extends 边构建"类 → 基类列表"映射，Phase 3 失败时沿继承链依次尝试；同时把 extends 边纳入 Phase 2 的 import 解析逻辑。4.1.2  多语言 Phase 2 基本不生效Phase 2 用 {imp}.{to_sym} 拼接去匹配符号名，此假设建立在 Python"模块路径=符号名前缀"的命名习惯上。对 tree-sitter 提取的 JS / TS / Go / Rust，imp 是文件路径或包名，与符号名命名空间完全不对应，跨文件调用链基本连不起来。改进方向：将 import 路径解析成实际文件路径（先支持相对路径），再在目标文件的符号里按短名匹配，本质上把按名字拼接，换为按文件定位再按名字查找。建议作为多语言支持的下一里程碑单独排期，短期先在 code_status 里标注"非 Python 语言的调用链解析能力有限"。4.1.3  轻量级局部类型推断obj = SomeClass(); obj.method() 这种模式当前完全无法解析。可在函数体扫描里加入简单的"同函数作用域内赋值追踪"：遇到 Assign(target=Name, value=Call) 就记录变量名 → 类型名映射，之后遇到 变量名.method() 时直接写成 类型名.method。此方法不追求完备，但覆盖"实例化后立即使用"这一最高频场景。4.2  P2  索引内容扩展4.2.1  模块级常量 / 类型别名不被索引_extract_python 只索引 FunctionDef / AsyncFunctionDef / ClassDef + import，模块级 Assign / AnnAssign（含类型注解赋值，常用于常量和 dataclass 字段）均不成为可被 code_explore 直接定位的符号。建议给其增加 kind="variable" / "constant" 的符号提取。4.2.2  FTS5 不含 doc 字段FTS5 建表只有 fts5(name, file, signature)，docstring 未被索引。对中文自然语言查询（code_explore 的核心入口）影响最大：目标函数的 docstring 里写了关键词但函数名里没有时，FTS5 查询命中不了。建议把 doc 字段加入 sym_fts，并在索引侧对中文做 2-gram 切分（与查询侧 _tokenize_query 对齐）。4.2.3  tree-sitter 路径 method / function 不区分_walk_ts 里所有函数类节点统一标为 kind: "function"，未区分 method（嵌套在类作用域内）。建议 scope 非空时标记为 "method"，与 Python 路径行为一致，避免 kind:method 过滤器在多语言场景下不生效。4.3  P3  查询体验增强全局重要性排名：在 index() 收尾阶段批量计算每个符号的 in_degree（被 calls / triggers / extends 指向次数）存为额外列，用于更稳定的排序，并支撑"项目最核心符号"查询。未被引用符号检测：基于 in_degree == 0（排除 entrypoint / dunder）辅助识别死代码，需将框架入口点（FunctionTool.call）排除以避免误判。模块公共接口查询：给定文件/模块名，返回其所有 public 符号及被哪些文件引用，是理解陌生模块的高频需求。4.4  P4  性能与可扩展性批量插入：index() 目前逐行 execute INSERT，改为按文件收集后 executemany 批量插入，对大仓库有明显收益。FTS5 增量重建：当前即使增量索引也全量重建 FTS5，改为只对发生变化的文件做增量 DELETE + INSERT INTO sym_fts，让"增量"在 FTS5 这一步也名副其实。用户自定义忽略规则：支持读取项目根目录下的 .codegraphignore（gitignore 语法），让用户排除生成代码、vendored 依赖等噪音。编辑后索引过期提示：safe_edit / multi_edit 成功写入文件后，如该文件在 .codegraph 索引范围内，附加建议"运行 code_index(incremental=true)"的提示。5  自我复核：误判与措辞修正在出具完初稿后，对所有关键论断逐一用 Python 脚本重新验证。发现 1 处措辞不精确：初稿表述更正后表述source_truncated 与 total_lines 字段被 _row_to_dict .split("\n")[0]，只保留类定义首行（如 class Foo(Bar):），完整的类体不会存入 DB。这意味着 code_explore 搜索一个 class 时 source 只有一行，LLM 无法从中得知类的属性、字段或 docstring 的完整内容。建议：与 FunctionDef 的处理方式对齐，存入完整类体（提高 [:500] 上限后对 class 同样适用），或至少存入首部 N 行（含 __init__ 签名和 docstring）。6.2  test_blocks_dotdot_traversal 重复定义tests/test_file_remove.py 第 76 行与第 88 行各定义一个 test_blocks_dotdot_traversal，第二个静默覆盖第一个（Python 类中同名方法后者覆盖前者），相当于有一个测试用例从未被执行。两个定义内容几乎相同，是复制粘贴遗留的死代码，不影响测试结论，但需要清理。7  总结与优先级建议codegraph 模块整体设计扎实，是这个插件里工程深度最突出的部分。自研的 Python AST + SQLite FTS5 + BFS 管线、三阶段引用解析、结构化失败回退，均超出"AI
